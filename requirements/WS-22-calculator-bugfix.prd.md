# 软件需求说明书 (SRS) — 计算器 BUG 修复与规格细化 (Issue #WS-22)

## 1. 系统边界与概述 (System Boundary & Overview)
- **业务概述**：本需求旨在对已有的 Web 简易计算器进行功能细化与体验优化。主要包括完善物理键盘的按键支持、解决在 1920×1080 桌面分辨率下的页面滚动条问题（确保页面布局自适应且无滚动条溢出），以及修正单数字输入下按 Enter 键触发计算的非预期行为。
- **系统边界**：
  - **在册范围**：
    1. 物理键盘输入细化：在现有键盘支持基础上，强化并规范对 `+`、`-`、`*`、`/` 键的捕获、事件默认拦截、运算逻辑处理以及对应 UI 按键的激活状态高亮反馈。
    2. 1920×1080 页面自适应排版约束：针对 1920×1080 分辨率制定页面和子面板（如历史面板）的高宽缩放和滚动隔离约束，杜绝出现全局滚动条。
    3. 计算触发前置校验：重构 Enter 键与 `=` 键的触发判断机制，明确只有在输入表达式“完善”时才执行计算、写入历史记录和更新界面；若仅有单操作数（或无未决运算符和历史记录），则不予触发并保持界面现状。
  - **排除范围**：
    1. 科学计算功能（如三角函数、幂函数等）。
    2. 移动端和非 1920×1080 分辨率之外的极端分辨率布局优化（本案仅对 1920×1080 分辨率下的全局滚动条溢出问题作强制修复约束，其他端继续沿用已有响应式逻辑）。
    3. 历史记录在云端的数据同步与账号体系。

## 2. 数据流拓扑 (Data Flow Topology)
- **数据流描述**：
  用户通过物理键盘或界面按钮输入字符。输入流经有效性校验后存入 Token 缓存区。当用户触发“确认/等号/Enter”时，系统新增“完整性校验”网关。若通过校验（即表达式完善），则送入计算引擎进行计算，并保存至 LocalStorage 与输出至 UI 显示；若未通过校验，则直接阻断数据流，界面及缓存均不作任何变化。
- **核心数据流向图/流程**：
```
  [物理键盘/点击输入] -> (输入有效性校验) -> [Token缓存区 (tokens / current)]
                                                  |
                                             (按 Enter / =)
                                                  |
                                        <是否输入完善？>
                                         /            \
                                    [是] /              \ [否]
                                        /                \
                    (计算引擎求值与高精度校正)          (阻断：无反应，不计入历史，无界面变化)
                                       |
                   [更新显示屏幕 / 写入 LocalStorage 历史记录]
```

## 3. 功能性需求 (Functional Requirements)

### 3.1 物理键盘输入细化 (Physical Keyboard Input)

#### REQ-KEYB-001: 物理操作符按键监听与映射
- **优先级**：高
- **前置条件**：浏览器窗口处于激活且计算器页面聚焦状态。
- **业务逻辑**：
  1. 系统必须监听全局 `keydown` 事件，捕获物理键盘的 `+`、`-`、`*` (或 `x`, `X`)、`/` 键。
  2. 触发按键时，必须调用 `event.preventDefault()`，防止浏览器执行默认行为（如 Firefox 按 `/` 键会触发快速查找，某些系统组合键可能缩放页面）。
  3. 将捕获的按键输入映射到计算器的操作符：
     - `+` 键映射为加法操作符（`+`）
     - `-` 键映射为减法操作符（`−`）
     - `*`、`x`、`X` 键映射为乘法操作符（`×`）
     - `/` 键映射为除法操作符（`÷`）
  4. 操作符输入缓冲规则：
     - 若当前输入区（`current`）为 `0` 且 Token 缓存区为空，按操作符键应将 `0` 作为第一操作数，并拼接该操作符（如 `0 +`）。
     - 若 Token 缓存区最后一个 Token 为操作符，且当前输入区为空，按新操作符键应直接替换上一个操作符。
- **输入数据流**：物理键盘按键事件（`+`, `-`, `*`, `x`, `X`, `/`）。
- **输出数据流**：更新 Token 缓存区状态，并在显示屏的公式区同步展示对应操作符。

#### REQ-KEYB-002: 键盘按键的 UI 视觉反馈
- **优先级**：中
- **前置条件**：用户通过物理键盘输入有效的计算器按键（包括数字、操作符、退格、全清、Enter键）。
- **业务逻辑**：
  1. 当监听到物理键盘的按键输入时，系统需根据映射关系定位至界面上的对应虚拟按键元素。
  2. 向该虚拟按键动态添加激活类名（例如 `.pressed` 或 `.active`），模拟用户点击该按钮时的按压阴影变化、缩放或背景色高亮动画。
  3. 在短时间内（如 100ms - 150ms）或在 `keyup` 事件触发时，移除激活状态类名，恢复原状，确保高亮反馈敏捷、不粘滞。
- **输入数据流**：键盘 `keydown` 及 `keyup` 事件。
- **输出数据流**：计算器虚拟按键的 CSS Class 状态变更与重绘。

### 3.2 计算触发规则优化 (Calculation Triggering)

#### REQ-CALC-001: 单操作数/未完善输入 Enter 拦截规则
- **优先级**：高
- **前置条件**：用户按下物理 `Enter` 键或点击界面的 `=` 按键。
- **业务逻辑**：
  1. 当接收到计算触发指令时，系统须先对当前 Token 缓存区（`tokens`）和当前输入（`current`）进行完整性评估。
  2. **拦截条件 1（单数字输入）**：
     - 若 Token 缓存区为空，且当前仅输入了一个数字（例如：屏幕只显示 `123`），此时触发计算属于“未完善输入”。
     - 系统必须拦截该计算请求。不调用计算引擎，不生成历史记录，公式显示区和主结果显示区不发生任何改变，保持原样。
  3. **拦截条件 2（单数字+操作符输入）**：
     - 若 Token 缓存区仅包含 `[操作数, 运算符]`（例如：屏幕显示 `123 +` 且没有当前输入数），并且当前不存在“上一次计算操作记录”（`lastOperation` 为空）。
     - 此时触发计算同样判定为“未完善输入”，系统须拦截该计算请求，界面和数据状态保持原样，不发生任何改变。
  4. **放行条件 1（输入完善的普通表达式）**：
     - 若表达式符合 `[操作数1, 运算符, 操作数2]`（如 `123 + 45`），属于“完善输入”，系统需执行计算，输出结果，并将 `123 + 45 = 168` 计入历史记录。
  5. **放行条件 2（重复计算逻辑/等号连击）**：
     - 若当前只存在一个操作数（如 `168`），但系统存在“上一次计算操作记录”（例如：`lastOperation` 为 `{ operator: "+", value: "45" }`），此时按 Enter 属于合法的等号重复计算，应放行。系统应执行 `168 + 45 = 213` 计算，并输出结果，同时将此新计算计入历史记录。
- **输入数据流**：计算触发事件、Token 缓存区状态、`lastOperation` 缓存状态。
- **输出数据流**：
  - 若拦截：无输出（数据 and 界面无变化）。
  - 若放行：输出计算结果至主屏幕，同时生成历史记录写入 LocalStorage。

### 3.3 界面排版与布局自适应 (Responsive Layout & Viewport)

#### REQ-LAYOUT-001: 1920×1080 分辨率无全局滚动条约束
- **优先级**：高
- **前置条件**：页面在分辨率为 1920×1080 的设备/浏览器视口下渲染。
- **业务逻辑**：
  1. **全局布局高度硬性限制**：
     - 根元素（`html`、`body`）的高度必须限制为 `100vh`（或 `100dvh`），并且设置 `overflow: hidden;`，从根本上防止因容器溢出产生浏览器垂直或水平滚动条。
     - 主容器（`.app-shell`）必须适应视口高度，其最大高度限制在 `100vh` 内，内边距和外边距使用合理的自适应单位或固定小数值。
  2. **弹性工作区高度分配**：
     - 工作区容器（`.workspace`）应采用弹性布局（Flexbox / Grid），高度充满除头部工具栏外的剩余视口空间（`flex: 1; min-height: 0;`）。
     - 计算器卡片（`.calculator-card`）与历史面板（`.history-panel`）高度必须自适应拉伸或限制在工作区高度内，不可超出 `.workspace`。
  3. **历史记录面板滚动隔离**：
     - 当历史记录项增多时，其高度溢出必须限制在历史记录面板内部。
     - 历史面板主体（`.history-panel`）的 `min-height` 设定必须与视口高度协调（例如，取消硬编码的 `min-height: 520px` 导致小屏幕溢出，改用百分比或自适应弹性高度限制），并设定最大高度，其子容器 `.history-list` 必须配置 `overflow-y: auto;` 滚动条，实现局部滚动。
  4. **排版缩放与防折行**：
     - 针对 1920×1080 桌面视图，计算器卡片各按键的高度、间距及字号需保持固定或使用 `clamp` 确保合理的紧凑视觉比例，防止大字号或大边距顶开父级容器导致内容截断或溢出。
- **输入数据流**：视口尺寸变化事件（`resize`）、浏览器渲染树。
- **输出数据流**：完全受限于 1920×1080 单屏显示内的自适应布局，不产生外层滚动条，只有历史面板内部支持纵向滚动。

## 4. 非功能性需求 (Non-Functional Requirements)

- **4.1 兼容性与默认事件阻断 (Event Prevention)**：
  - 必须完整处理所有目标按键在不同浏览器（Chrome、Firefox、Safari、Edge）下的默认按键事件冲突，保证 `keydown` 全局监听的可靠性。
- **4.2 布局健壮性与防变形 (Layout Robustness)**：
  - 在 1920x1080 分辨率下，即使浏览器全屏或视口稍作缩放，计算器主卡片 and 历史记录侧边栏的大小及间距也应保持美观比例，不发生重叠或挤压变形。
- **4.3 零输入漏失性能 (Performance)**：
  - 键盘监听器对 `+`、`-`、`*`、`/` 键的反应延迟必须控制在 16ms（即 1 帧）以内，保证用户的物理键盘打字体验无任何卡顿。
