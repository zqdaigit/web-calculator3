# 软件需求说明书 (SRS) — Web 简易计算器 (Issue #WS-16)

## 1. 系统边界与概述 (System Boundary & Overview)
- **业务概述**：本需求旨在设计并实现一个跨平台、美观且响应迅速的 Web 端简易计算器。该计算器能够自适应运行在手机、平板和 PC 等不同设备的浏览器中，提供便捷的日常算术计算服务，并具有优良的用户交互体验。
- **系统边界**：
  - **在册范围**：
    1. 计算核心逻辑：支持基础四则运算（加、减、乘、除）以及单目运算（百分比、正负号切换、退格、清空）。
    2. 用户界面：响应式设计（自适应手机、平板与 PC），支持物理键盘输入映射，具备深/浅色主题切换。
    3. 数据存储：支持本地历史计算记录的存储与展示（基于 LocalStorage），用户可点击历史记录复用结果或表达式。
    4. 输入校验：计算缓冲区输入校验（如连续运算符输入处理、多个小数点校验、除零错误控制、数据精度截断）。
  - **排除范围**：
    1. 复杂科学计算：不包含微积分、矩阵运算、复数运算及科学函数（如三角函数 sin/cos、对数 log 等）。
    2. 多端同步与云存储：不包含用户账户注册、云端数据同步及跨设备历史记录备份，所有数据仅存储于本地浏览器。
    3. 原生 App 打包：不包含打包为 iOS/Android 原生应用的封装逻辑。

## 2. 数据流拓扑 (Data Flow Topology)
- **数据流描述**：
  用户通过界面按键或物理键盘输入计算指令，系统接收输入流后进行有效性校验（如防止重复输入小数点、处理多余运算符等），并将其缓存至输入缓冲区。用户点击“等号”或按下物理 Enter 键时，计算引擎对缓冲区中的公式进行求值计算，经过浮点数高精度校正后将结果输出到显示屏幕，同时将当前的“表达式 + 结果”序列化保存到本地浏览器 LocalStorage。
- **核心数据流向图/流程**：
  ```
  [用户界面按键 / 物理键盘输入] -> (输入有效性校验与缓冲区暂存) -> [输入缓冲区/显示公式]
  [输入缓冲区/显示公式] -> (计算引擎求值与浮点精度校正) -> [结果显示区] -> (序列化保存) -> [LocalStorage] -> (读取展现) -> [历史记录侧边栏/面板]
  ```

## 3. 功能性需求 (Functional Requirements)

### 3.1 核心计算服务 (Core Calculation Service)

#### REQ-CALC-001: 基础四则运算
- **优先级**：高
- **前置条件**：系统已正常加载并处于待机/输入状态。
- **业务逻辑**：
  1. 用户点击数字按键（0-9）及运算符按键（+、-、*、/）组合成表达式。
  2. 点击 '=' 或按物理键盘 Enter 键触发计算。
  3. 系统解析表达式，遵循标准数学运算法则（先乘除后加减，从左到右）计算结果。
- **输入数据流**：用户输入的数字字符序列与操作符序列。
- **输出数据流**：计算结果展示在结果屏幕上，旧公式显示在公式栏中。

#### REQ-CALC-002: 单目运算与特殊操作
- **优先级**：中
- **前置条件**：计算器有当前输入值或上一次计算结果。
- **业务逻辑**：
  1. **百分比 (%)**：将当前显示的数值乘以 0.01。
  2. **正负号切换 (+/-)**：将当前数值的符号取反（正数变负数，负数变正数）。
  3. **清除键 (AC)**：清空整个计算缓冲区、当前显示状态及当前的内存运算状态（历史记录不受影响）。
  4. **退格/删除键 (C / Backspace)**：删除当前输入的最后一个字符。
- **输入数据流**：用户点击 `%`、`+/-`、`AC` 或 `C` 的控制指令。
- **输出数据流**：更新后的显示屏数值或输入缓冲区状态。

#### REQ-CALC-003: 浮点数高精度校正与溢出校验
- **优先级**：高
- **前置条件**：触发计算引擎进行求值。
- **业务逻辑**：
  1. 为避免 JavaScript 默认的二进制浮点数精度偏差（如 `0.1 + 0.2 = 0.30000000000000004`），计算引擎在输出结果前需进行十进制校正逻辑，保留最多 12 位有效数字。
  2. 校验计算结果长度，若超出屏幕显示的最大范围（如 15 位数字），需转换为科学计数法（Scientific Notation）展示。
- **输入数据流**：计算求值后的原始浮点数。
- **输出数据流**：校正后的十进制精确字符串，或转换为科学计数法后的字符串。

#### REQ-CALC-004: 连续/链式计算
- **优先级**：高
- **前置条件**：缓冲区中已存在操作数与运算符。
- **业务逻辑**：
  1. 当缓冲区已存在一个完整运算（如 `1 + 2`），用户不按 `=` 而是直接按下运算符（如 `+`），系统应自动计算前两个数的中间结果（即 `3`），并将 `3` 作为新的操作数，拼接当前的运算符（如 `+`），等待后续输入。
  2. 在按下 `=` 获得结果后，如果用户直接点击运算符（如 `*`），应自动将当前结果作为第一个操作数，继续进行链式计算。
- **输入数据流**：在有未决计算的状态下输入的新运算符。
- **输出数据流**：中间结果显示在主屏幕，公式栏更新为 `[中间结果] [新运算符]`。

### 3.2 历史记录服务 (History Service)

#### REQ-CALC-005: 历史记录管理
- **优先级**：中
- **前置条件**：用户完成了一次完整的等号计算，并产生了有效的“表达式 = 结果”对。
- **业务逻辑**：
  1. 系统在每次点击等号并计算成功后，将当前的“运算公式”和“计算结果”作为一个结构化对象存入 LocalStorage。
  2. 历史记录限制最大条数（如 50 条），超出则按照 FIFO（先进先出）原则淘汰旧记录。
  3. 用户可以通过侧边栏或滑动面板查看历史计算记录。
  4. 支持用户点击单条历史记录，将其对应的“结果”或整个“表达式”重新载入当前的计算缓冲区。
  5. 提供“清空历史记录”功能，点击后清空 LocalStorage 和历史界面。
- **输入数据流**：保存指令（自动触发）、点击载入指令、清空指令。
- **输出数据流**：历史记录面板刷新，LocalStorage 状态变更，计算缓冲区更新。

### 3.3 交互与环境支持 (Interactive Support)

#### REQ-CALC-006: 物理键盘映射支持
- **优先级**：中
- **前置条件**：页面处于激活聚焦状态。
- **业务逻辑**：
  1. 系统监听全局 `keydown` 事件。
  2. 按键映射表：
     - 数字 `0-9` 与小数点 `.` 映射为对应输入。
     - `+`, `-`, `*` (或 `x`), `/` 键映射为加、减、乘、除运算符。
     - `Enter` 或 `=` 键映射为触发计算。
     - `Backspace` 键映射为退格（删除最后一个字符）。
     - `Escape` 或 `Delete` 键映射为 `AC`（全清）。
  3. 触发映射时，对应的虚拟界面按键需产生按压高亮动画，提供视觉反馈。
- **输入数据流**：键盘按键事件。
- **输出数据流**：更新缓冲区，并触发按键高亮样式。

## 4. 非功能性需求 (Non-Functional Requirements)
- **4.1 响应式适配 (Responsiveness)**：
  - **手机端（Mobile）**：采用单列竖屏布局，按键间距适中，防止误触。隐藏不常用的历史面板，改用滑动或折叠按钮触发。
  - **平板端（Tablet）**：支持横竖屏自适应，历史记录可作为分栏固定在侧边或通过浮窗展示。
  - **PC端（Desktop）**：主界面居中，按键支持悬停（Hover）特效；历史记录分栏常驻右侧，提供更宽广的视图。
- **4.2 界面风格与美观性 (UI/UX Aesthetics)**：
  - 使用现代扁平化、新拟物化（Neumorphism）或毛玻璃（Glassmorphism）设计风格，提供优雅的配色。
  - 必须支持“深色模式 (Dark Mode)”与“浅色模式 (Light Mode)”，主题切换有平滑过渡动画（过渡时间 300ms）。
  - 按键点击应伴有轻微的下沉感、阴影变化或微交互效果。
- **4.3 性能与加载 (Performance)**：
  - 纯前端实现，无外部接口依赖，首屏加载时间在常规 3G 网络下应小于 1.5s。
  - 每次按键计算延迟小于 50ms，无卡顿。
- **4.4 健壮性与错误处理 (Robustness & Error Handling)**：
  - **除零错误**：当出现 `n / 0` 时，主屏幕显示 "Error" 或 "Cannot divide by zero"，并冻结除 `AC` 外的其他输入，直到用户清空重试。
  - **非法表达式校验**：系统应在前端阻止非法输入组合（例如 `++`, `.*` 等），并在求值失败时优雅捕获异常，显示 "Error" 而非导致前端崩溃。
  - **字号自动缩放**：随着显示屏数值字符长度增加，字体大小应自适应缩小（例如从 3rem 渐变到 1.5rem），防止字符溢出显示边界。

## 5. 核心业务概念实体 (Core Business Entities)
- **5.1 操作数 (Operand)**：参与运算的数值，可以是整数、浮点数，支持正负数表示。
- **5.2 运算符 (Operator)**：加(+)、减(-)、乘(*)、除(/)等双目算术运算符，以及百分比(%)、正负号(+/-)等单目运算符。
- **5.3 计算缓冲区 (Calculation Buffer)**：用于临时存放当前用户输入但尚未求值的表达式序列，包含操作数和运算符。
- **5.4 历史记录项 (History Item)**：
  - 结构定义：
    ```json
    {
      "id": "String (UUID/Timestamp)",
      "expression": "String (e.g., '12 + 45')",
      "result": "String (e.g., '57')",
      "timestamp": "Number (UTC Epoch Milliseconds)"
    }
    ```
- **5.5 界面主题 (UI Theme)**：系统支持的主题状态枚举：`Light`（浅色模式）和 `Dark`（深色模式）。

## 6. 用户故事主线 (User Story Mainline)

### 6.1 用户故事 1：日常简单算术与结果复用
- **场景**：小张在电脑上做账，需要算一笔数。
- **流程**：
  1. 小张在 PC 浏览器打开计算器，默认为浅色模式。小张觉得刺眼，点击右上角太阳/月亮图标切换为深色模式。
  2. 他通过小键盘输入 `1024 + 768`，屏幕上方的公式显示区显示 `1024 + 768`，主显示区显示 `1792`。
  3. 他按回车键，计算器得出结果 `1792`。同时，右侧历史记录面板新增一条记录：`1024 + 768 = 1792`。
  4. 接着小张需要算另一笔账：`1792 * 1.2`。他发现 `1792` 是刚才的历史结果，于是他点击历史记录面板里的 `1792`，该数字被自动填入当前的输入框。
  5. 他输入 `* 1.2` 并按回车，得到 `2150.4`。

### 6.2 用户故事 2：链式混合运算与退格纠错
- **场景**：小李在手机上算账，不小心按错了键。
- **流程**：
  1. 小李输入 `150 + 20`，本想按 `*`，但手指粗按成了 `/`。公式显示区显示 `150 + 20 /`。
  2. 小李发现按错后，点击了退格键 `C`（或虚拟删除键），退格键删除了最后的 `/`，屏幕恢复为 `150 + 20`。
  3. 小李重新点击 `*` 键，输入 `3`，屏幕显示 `150 + 20 * 3`。
  4. 小李点击 `=` 键，系统遵循先乘除后加减法则，输出结果 `210`。
